# 问题四：基于 OR-logic 的出厂水浊度风险评价模型报告

## 1. 建模目标

问题四要求以水浊度 `NTU` 为核心指标，结合超标幅度与异常持续时长，建立水质风险评价体系，并将 2026 年近 3 个月的水质状态划分为：

```text
安全、低风险、中风险、高风险
```

同时，需要给出各风险等级的天数占比，并列出 2026 年 3 月份每日的具体风险分类结果。

题目给出的统一硬性约束为：

```text
生活饮用水浊度限值 ≤ 1 NTU
```

因此，`1 NTU` 是本文判断超标、风险修正和最终分类的统一标准。

本题的建模目标不是简单判断某一天是否超标，而是要综合考虑：

```text
1. 是否超标；
2. 超标幅度有多大；
3. 超标持续了多久；
4. 水质是否波动剧烈；
5. 水质是否存在持续上升趋势；
6. 最终风险等级是否符合 1 NTU 国标硬约束。
```

因此，本文构建 **OR-logic 综合风险评价模型**。该模型首先将每日出厂水 `NTU` 序列转化为多个日尺度风险指标，然后将各指标归一化为 `[0, 1]` 区间内的风险证据，最后通过 soft-OR 逻辑融合得到连续综合风险分数，并结合 `1 NTU` 国标硬约束得到最终四级风险等级。

---

## 2. 数据对象与运行日定义

### 2.1 评价对象

本文以出厂水浊度作为风险评价对象，即数据中的：

```text
NTU
```

该指标对应清水/处理后水浊度，是问题四中水质风险评价的核心变量。

代码中设定：

```python
TARGET_COL = "NTU"
```

国标限值设定为：

```python
STANDARD_LIMIT = 1.0
```

即：

```text
NTU ≤ 1：达标
NTU > 1：超标
```

---

### 2.2 运行日定义

原始数据每 2 小时采样一次，每天理论上有 12 个观测点。但是水厂运行具有连续性，凌晨时段的数据通常仍然属于前一天的运行周期。如果直接按照自然日 `00:00–22:00` 划分，可能会把一个完整的水厂运行过程割裂成两天。

因此，本文定义水厂运行日为：

```text
当日 07:00 至次日 05:00
```

一个完整运行日理论上包含：

```text
07:00, 09:00, 11:00, 13:00, 15:00, 17:00,
19:00, 21:00, 23:00, 次日 01:00, 03:00, 05:00
```

共 12 个观测点。

在代码中，凌晨 7 点以前的数据会被归入前一天的运行日：

```python
data["OP_DATE"] = data["DATETIME"].dt.normalize()

before_seven = data["DATETIME"].dt.hour < 7

data.loc[before_seven, "OP_DATE"] = (
    data.loc[before_seven, "DATETIME"] - pd.Timedelta(days=1)
).dt.normalize()
```

这样可以保证一个运行周期内的水质变化不会被自然日边界切断。

---

### 2.3 有效点数要求

每个完整运行日理论上有：

```python
EXPECTED_POINTS_PER_DAY = 12
```

为了避免缺失数据导致误判，本文设置最小可评分点数：

```python
MIN_VALID_POINTS = 6
```

即：

```text
如果某个运行日有效 NTU 点数 ≥ 6，则计算风险评分；
如果某个运行日有效 NTU 点数 < 6，则认为数据不足，不计算风险评分。
```

这样处理的原因是：

```text
缺失数据不能被等价看作安全数据。
```

如果把缺失日直接归为安全或低风险，会低估真实风险。因此，对于有效数据不足的日期，模型保留该日期，但不强行给出风险等级。

## 3. 日尺度风险指标构造

对于每一个可评分运行日，本文构造五个日尺度风险指标：

```text
max_excess
exceed_ratio
median_excess
ntu_std
longest_rising_steps
```

这五个指标分别从峰值超标、持续超标、典型超标强度、日内波动和连续恶化趋势五个角度刻画水质风险。

---

### 3.1 最大超标幅度 `max_excess`

最大超标幅度定义为：

$$
\text{max\_excess}
=
\max(\text{NTU} - 1, 0)
$$

也就是当天所有观测点中超过 `1 NTU` 的最大幅度。

例如，若某日最大 NTU 为 `1.8`，则：

$$
\text{max\_excess}
=
1.8 - 1.0
=
0.8
$$

若全天所有 NTU 均不超过 `1`，则：

$$
\text{max\_excess} = 0
$$

该指标反映的是：

```text
峰值超标风险。
```

它回答的问题是：

```text
这一天最严重的时候超标了多少？
```

---

### 3.2 超标比例 `exceed_ratio`

超标比例定义为：

$$
\text{exceed\_ratio}
=
\frac{\#(\text{NTU} > 1)}
{\#(\text{valid NTU points})}
$$

其中：

```text
#(NTU > 1)：当天超标观测点数量；
#(valid NTU points)：当天有效 NTU 观测点数量。
```

例如，一天有 12 个有效观测点，其中 3 个点超过 1 NTU，则：

$$
\text{exceed\_ratio}
=
\frac{3}{12}
=
0.25
$$

该指标反映的是：

```text
超标持续性。
```

它回答的问题是：

```text
这一天有多大比例的时间处于超标状态？
```

---

### 3.3 超标中位幅度 `median_excess`

超标中位幅度只在超标点上计算：

$$
\text{median\_excess}
=
\operatorname{median}(\text{NTU} - 1 \mid \text{NTU} > 1)
$$

如果当天没有超标点，则：

$$
\text{median\_excess}=0
$$

这个指标与 `max_excess` 的区别在于：

```text
max_excess 关注最极端峰值；
median_excess 关注典型超标程度。
```

如果某一天只有一个极端异常尖峰，`max_excess` 会较高，但 `median_excess` 不一定高。加入 `median_excess` 可以避免风险评价完全被单个异常峰值控制。

该指标反映的是：

```text
典型超标强度。
```

---

### 3.4 日内波动程度 `ntu_std`

日内波动程度定义为当天有效 NTU 值的总体标准差：

$$
\text{ntu\_std}
=
\operatorname{std}(\text{NTU})
$$

代码中使用：

```python
valid_values.std(ddof=0)
```

即总体标准差。

即使某一天没有明显超标，如果 NTU 在一天内频繁波动，也说明水厂运行状态不够稳定，可能存在潜在风险。

该指标反映的是：

```text
水质稳定性风险。
```

---

### 3.5 最长连续上升步数 `longest_rising_steps`

最长连续上升步数用于刻画水质是否存在持续恶化趋势。

模型统计满足以下条件的最长连续上升段：

```text
1. 相邻两个观测点时间间隔必须正好为 2 小时；
2. 前后两个 NTU 都必须大于预警水平 0.7；
3. 后一个 NTU 必须大于前一个 NTU。
```

代码中预警水平设定为：

```python
WARNING_LEVEL = 0.7
```

即只有当水浊度已经接近 `1 NTU` 国标限值，并且连续上升时，才认为存在趋势性风险。

如果出现以下任一情况，连续上升计数会重置：

```text
1. 时间间隔不是 2 小时；
2. 前后任一值缺失；
3. 后一个 NTU 没有上升；
4. 前后任一 NTU ≤ 0.7。
```

该指标反映的是：

```text
异常发展趋势和潜在恶化风险。
```

它回答的问题是：

```text
水质是否正在连续变差？
```

## 4. 风险指标归一化

由于五个风险指标的量纲不同，不能直接相加或比较。因此，本文使用固定风险上限将每个指标归一化到 `[0, 1]` 区间。

归一化公式为：

$$
r_i
=
\min\left(
\frac{x_i}{c_i},
1
\right)
$$

其中：

```text
x_i = 第 i 个原始风险指标；
c_i = 第 i 个指标的固定风险上限；
r_i = 第 i 个归一化风险证据。
```

代码中使用的固定风险上限为：

| 指标 | 含义 | 风险上限 |
|---|---|---:|
| `max_excess` | 最大超标幅度 | 4.0 |
| `exceed_ratio` | 超标比例 | 0.50 |
| `median_excess` | 超标中位幅度 | 0.50 |
| `ntu_std` | 日内标准差 | 0.50 |
| `longest_rising_steps` | 最长连续上升步数 | 4.0 |

例如，如果某天：

```text
exceed_ratio = 0.25
```

则归一化后为：

$$
r_{\text{exceed}}
=
\frac{0.25}{0.50}
=
0.50
$$

如果某项指标超过上限，则截断为 1：

$$
r_i = 1
$$

这表示该维度的风险证据已经达到最高等级。

---

## 5. OR-logic 综合风险评分

五个归一化风险指标记为：

$$
r_1, r_2, r_3, r_4, r_5
$$

本文使用 soft-OR 逻辑将其融合为综合风险分数：

$$
R
=
1
-
\prod_{i=1}^{5}(1-r_i)
$$

即：

$$
R
=
1
-
(1-r_1)(1-r_2)(1-r_3)(1-r_4)(1-r_5)
$$

其中：

```text
R = OR_risk_score
```

该公式可以理解为：

```text
1 - 所有风险因素都不触发的可能性
```

因此，`R` 表示：

```text
至少一个风险因素触发的综合风险强度。
```

---

## 6. 使用 OR-logic 的原因

水质风险评价具有明显的“任一风险源触发即可提高风险”的特点。例如：

```text
1. 某天最大 NTU 很高，即使持续时间短，也应提高风险；
2. 某天峰值不高，但长时间接近或超过 1 NTU，也应提高风险；
3. 某天没有明显超标，但波动很大且连续上升，也应有预警风险；
4. 多个中等风险同时出现时，整体风险应被放大。
```

如果使用简单加权平均，可能出现一个高风险指标被其他低风险指标稀释的问题。

而 OR-logic 的特点是：

```text
1. 单个风险指标很高时，综合风险会明显升高；
2. 多个中等风险指标叠加时，综合风险也会升高；
3. 不需要人为设置主观权重；
4. 输出稳定落在 [0, 1] 区间；
5. 具有清晰的逻辑解释。
```

因此，OR-logic 更符合水质安全管理中的保守风险评价思想。

## 7. 风险等级初始分类

得到连续综合风险评分 `R` 后，模型首先根据评分阈值进行初始风险分类。

代码中实际使用的阈值为：

| OR 综合评分区间 | 初始风险等级 |
|---:|---|
| `R <= 0.10` | 安全 |
| `0.10 < R <= 0.40` | 低风险 |
| `0.40 < R < 0.99` | 中风险 |
| `R >= 0.99` | 高风险 |

即：

```python
RISK_BOUNDARIES = {
    "safe_low": 0.10,
    "low_moderate": 0.40,
    "moderate_high": 0.99,
}
```

分类函数为：

```python
def classify_by_score(score):
    if pd.isna(score):
        return pd.NA
    if score <= 0.10:
        return "安全"
    if score <= 0.40:
        return "低风险"
    if score < 0.99:
        return "中风险"
    return "高风险"
```

需要注意：

```text
论文中所有阈值必须与代码保持一致。
```

如果后续修改阈值，则 notebook、结果表和论文表述都需要同步修改。

## 8. 国标 NTU 硬约束修正

题目明确给出：

```text
生活饮用水浊度限值 ≤ 1 NTU
```

因此，仅依靠 OR 综合评分进行分类还不够。本文在初始分类之后加入国标硬约束，保证最终等级符合题目要求。

---

### 8.1 若当日最大 NTU > 1

如果某日：

$$
\max(\text{NTU}) > 1
$$

说明该运行日已经发生超标。

因此最终风险等级不得低于：

```text
中风险
```

如果初始等级为：

```text
安全
```

或：

```text
低风险
```

则强制上调为：

```text
中风险
```

如果初始等级已经是：

```text
中风险
```

或：

```text
高风险
```

则保持不变。

---

### 8.2 若当日最大 NTU < 1

如果某日：

$$
\max(\text{NTU}) < 1
$$

说明全天所有有效观测点均未超过国标限值。

因此最终风险等级不得高于：

```text
低风险
```

如果初始等级为：

```text
中风险
```

或：

```text
高风险
```

则强制下调为：

```text
低风险
```

这样处理的含义是：

```text
未超标但波动较大或存在上升趋势的日期，可以作为低风险预警；
但不应直接判定为中风险或高风险。
```

---

### 8.3 若当日最大 NTU = 1

如果：

$$
\max(\text{NTU}) = 1
$$

则刚好处于国标边界。

模型不额外调整风险等级，保持初始分类结果。

---

### 8.4 缺失或有效点不足日期

对于：

```text
无观测记录
```

或：

```text
有效点数 < 6
```

的日期，模型保持未分类状态，不强行归为安全。

原因是：

```text
缺失数据无法证明水质安全。
```

## 10. 模型整体流程

本模型的完整流程如下：

```text
原始 2 小时 NTU 数据
        ↓
构造运行日：当日 07:00 至次日 05:00
        ↓
统计每日有效 NTU 点数
        ↓
若有效点数 < 6：
    保留日期，但不计算风险评分
        ↓
若有效点数 ≥ 6：
    计算五维日风险指标
        ↓
五维指标：
    max_excess
    exceed_ratio
    median_excess
    ntu_std
    longest_rising_steps
        ↓
固定上限归一化：
    r_i = min(x_i / c_i, 1)
        ↓
OR-logic 综合评分：
    R = 1 - ∏(1 - r_i)
        ↓
按 OR 分数初始分类：
    安全 / 低风险 / 中风险 / 高风险
        ↓
施加 NTU ≤ 1 国标硬约束：
    超标日不得低于中风险
    未超标日不得高于低风险
        ↓
得到最终风险等级 final_risk
        ↓
输出：
    近 3 个月等级天数占比
    3 月每日风险等级
    约束修正记录
    Excel 结果表
```

## 13. 模型合理性分析

### 13.1 同时考虑超标幅度和持续时间

题目要求结合超标幅度与异常持续时长。

本文中的：

```text
max_excess
median_excess
```

用于刻画超标幅度；

```text
exceed_ratio
```

用于刻画超标持续性；

```text
longest_rising_steps
```

用于刻画持续上升趋势。

因此模型与题目要求一致。

---

### 13.2 避免只依赖单一最大值

如果只看最大 NTU，则某一天的单个异常尖峰可能导致风险被过度放大。

本文加入：

```text
exceed_ratio
median_excess
ntu_std
longest_rising_steps
```

可以同时反映：

```text
持续性
典型超标程度
波动性
趋势性
```

从而避免单指标评价的片面性。

---

### 13.3 OR-logic 符合安全风险评价思想

水质安全问题具有保守性。

只要某一个风险维度明显异常，就应提高整体风险等级。

OR-logic 的公式：

$$
R = 1 - \prod_i(1-r_i)
$$

能够体现：

```text
任一风险源触发即可提高综合风险
```

的逻辑，因此比简单平均更适合风险评价。

---

### 13.4 国标硬约束保证分类合规

题目给定 `1 NTU` 为统一标准。

因此本文不是单纯根据数据分布分类，而是在最终等级中加入：

```text
max_NTU > 1：最终等级不得低于中风险
max_NTU < 1：最终等级不得高于低风险
```

这保证了模型结果符合题目的硬性要求。

---

### 13.5 缺失日不被误判为安全

模型要求有效点数至少为 6。

如果数据不足，则保持未分类。

这避免了：

```text
缺失数据被当作安全数据
```

的错误。

## 14. 模型局限性

本模型也存在一定局限性。

第一，风险上限参数和风险等级阈值具有一定人工设定成分。例如：

```text
max_excess cap = 4.0
exceed_ratio cap = 0.50
safe/low boundary = 0.10
low/moderate boundary = 0.40
```

这些参数基于题目约束、数据分布和风险解释需求设定。若用于实际工程系统，需要结合水厂运行经验进一步校准。

第二，OR-logic 模型没有直接建模各指标之间的因果关系，而是将它们视为多个风险证据进行融合。因此，它更适合作为风险评价模型，而不是机理预测模型。

第三，对于数据缺失较多的日期，模型不进行评分。这保证了风险评价的严谨性，但也会导致部分日期无法分类。

## 15. 可直接写入论文的方法段

本文针对问题四构建了基于 OR-logic 的出厂水浊度风险评价模型。首先，将原始 2 小时采样的 `NTU` 数据按照水厂运行周期重新组织，将一个运行日定义为当日 07:00 至次日 05:00。每个完整运行日理论上包含 12 个观测点。对于有效 NTU 点数少于 6 个的日期，模型认为数据不足，不计算风险评分，以避免将缺失数据误判为低风险或安全。

对于每一个可评分运行日，本文构造五个日尺度风险指标：最大超标幅度、超标比例、超标中位幅度、日内 NTU 标准差以及最长连续上升步数。其中，最大超标幅度用于反映峰值超标风险，超标比例用于刻画异常持续性，超标中位幅度用于衡量典型超标强度，日内标准差反映水质波动性，最长连续上升步数用于描述水质持续恶化趋势。随后，将各指标按照固定风险上限归一化至 `[0,1]` 区间。

设归一化后的五个风险指标为 `r_i`，本文采用 soft-OR 逻辑计算综合风险评分：

$$
R = 1 - \prod_i (1-r_i)
$$

该评分可以解释为至少一个风险因素被触发的综合风险强度。与简单加权平均相比，该方法能够避免单个高风险指标被其他低风险指标稀释，更符合水质安全评价中“任一风险源异常均需预警”的保守原则。

根据综合风险评分，本文首先划分初始风险等级：当 `R <= 0.10` 时判为安全，`0.10 < R <= 0.40` 时判为低风险，`0.40 < R < 0.99` 时判为中风险，`R >= 0.99` 时判为高风险。随后结合生活饮用水浊度限值 `1 NTU` 施加硬约束：若当日最大 NTU 大于 1，则最终等级不得低于中风险；若当日最大 NTU 小于 1，则最终等级不得高于低风险；若最大 NTU 等于 1，则不进行额外调整。最终得到 2026 年近 3 个月的风险等级占比以及 3 月份每日风险分类结果。
